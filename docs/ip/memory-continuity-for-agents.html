Memory Continuity for Long-Running Agents

Summary

This draft establishes prior art around agent memory continuity at a category level. It does not reveal private memory files, identity doctrine, scars, relationship context, internal prompts, or production recovery rituals.

Thesis

Long-running agents need more than chat history.

A useful agent must be able to preserve what it is doing, why it is doing it, what step comes next, what is blocked, and what has been completed but not yet reported.

This is working-state continuity, distinct from long-term memory.

The Problem

Agent sessions are fragile. They can be interrupted by context compaction, model fallback, process restart, tool failure, network interruption, user reset, and long-running task boundaries.

If the only record is a transcript, the next session must reconstruct intent from noisy history. That is slow, error-prone, and often incomplete.

Core Pattern

A continuity-aware agent runtime separates at least three layers:

  1. Transcript history — what was said.
  2. Long-term memory — durable facts, preferences, decisions, and lessons.
  3. Current working state — the active objective, current step, next action, blockers, and unsurfaced results.

The third layer is the key. It is short, structured, and overwritten as work changes.

Minimal Working-State Schema

A public-safe schema can include:

Objective: what the agent is trying to accomplish
Current Step: what was last completed or is active now
Key Decisions: current relevant decisions only
Next Action: exactly what should happen next
Blockers: what prevents progress
Unsurfaced Results: completed outputs not yet shown to the user
Last Updated: timestamp

This schema is intentionally generic. It does not require revealing private doctrine or user data.

Why This Matters

Working-state continuity enables recovery after interruption, safer handoffs, fewer repeated mistakes, clearer user trust, better long-running task execution, and reduced hallucination about prior work.

Defensive Claim

The category established here is:

A memory continuity layer for long-running agents that separates transcript history, long-term memory, and overwrite-oriented current working state containing objective, current step, decisions, next action, blockers, and unsurfaced results.

Boundaries

  • No private memory files.
  • No real user facts.
  • No relationship histories.
  • No identity/soul doctrine.
  • No compaction scars.
  • No internal recovery rituals.
  • No production memory paths.
  • No model routing doctrine.

Public Example

A safe toy example:

A coding agent begins a refactor, writes a current-state checkpoint before running tests, gets interrupted, and later resumes from the checkpoint without rereading the entire conversation.

Closing

Agents that cannot remember what they are doing cannot be trusted with long-running work. Continuity is not a cosmetic feature. It is runtime infrastructure.
